# AI 에이전트 답변 검증 체크리스트

> 이 문서는 AI 에이전트와 함께 개발을 진행할 때 답변 내용을 검증하기 위한 가이드입니다.
> AI 에이전트의 답변이 100% 완벽하지 않을 수 있으므로, 항상 아래 체크리스트를 통해 검증해야 합니다.

## 1. 정보 정확성 검증

### 1.1 거짓 정보 확인

- [ ] 제공된 정보가 실제로 존재하는 라이브러리, API, 기능인지 확인하세요
- [ ] 버전별 기능 차이가 정확히 반영되었는지 확인하세요
- [ ] 공식 문서와 다른 내용이 있는지 확인하세요
- [ ] 더 이상 지원되지 않는(deprecated) 기능을 사용하지 않았는지 확인하세요

### 1.2 환각(Hallucination) 검증

- [ ] 존재하지 않는 함수, 메서드, 속성을 사용하지 않았는지 확인하세요
- [ ] 임의로 만들어낸 것으로 보이는 패키지명이나 경로가 없는지 확인하세요
- [ ] 실제 프로젝트에 존재하지 않는 파일이나 폴더를 참조하지 않았는지 확인하세요

## 2. 코드 중복성 검증

### 2.1 기존 코드 재활용 확인

- [ ] 이미 존재하는 비즈니스 로직을 불필요하게 재생성하지 않았는지 확인하세요
- [ ] 기존 유틸리티 함수, 헬퍼 함수를 활용할 수 있는지 확인하세요
- [ ] 프로젝트에 이미 설치된 라이브러리로 해결 가능한지 확인하세요
- [ ] 동일한 기능을 하는 컴포넌트가 이미 존재하는지 확인하세요

### 2.2 코드 일관성 확인

- [ ] 기존 코드와 유사한 패턴으로 작성되었는지 확인하세요
- [ ] 같은 기능을 다른 방식으로 구현하지 않았는지 확인하세요

## 3. 오버 엔지니어링 검증

### 3.1 필요성 확인

- [ ] 요구사항에 없는 기능을 추가하지 않았는지 확인하세요
- [ ] 과도하게 복잡한 추상화를 사용하지 않았는지 확인하세요
- [ ] 현재 요구사항에 맞는 적절한 수준의 구현인지 확인하세요
- [ ] 미래의 확장성을 위해 지나치게 복잡하게 설계하지 않았는지 확인하세요

### 3.2 단순성 원칙

- [ ] YAGNI(You Aren't Gonna Need It) 원칙을 위배하지 않았는지 확인하세요
- [ ] KISS(Keep It Simple, Stupid) 원칙을 준수했는지 확인하세요
- [ ] 더 간단한 방법으로 같은 결과를 얻을 수 있는지 확인하세요

## 4. 테스트 검증

### 4.1 테스트 우회 확인

- [ ] 테스트를 통과하기 위해 부정직한 방법을 사용하지 않았는지 확인하세요
- [ ] 테스트 자체를 무력화시키는 코드가 없는지 확인하세요
- [ ] 모킹(Mocking)을 과도하게 사용하여 실제 로직을 테스트하지 못하는 상황이 아닌지 확인하세요
- [ ] 하드코딩된 값으로 테스트를 통과시키지 않았는지 확인하세요

### 4.2 테스트 품질 확인

- [ ] 테스트가 실제로 요구사항을 검증하는지 확인하세요
- [ ] 엣지 케이스(edge case)를 적절히 처리하는지 확인하세요
- [ ] 테스트 커버리지가 적절한지 확인하세요

## 5. 코드 스타일 및 아키텍처 검증

### 5.1 린팅 및 포맷팅

- [ ] 기존 프로젝트의 ESLint 규칙을 준수했는지 확인하세요
- [ ] Prettier 설정에 맞게 포맷팅되었는지 확인하세요
- [ ] 프로젝트의 명명 규칙(naming convention)을 따르는지 확인하세요
- [ ] `absolute-rule.md`에 정의된 CSS 포맷팅 규칙을 준수했는지 확인하세요

### 5.2 아키텍처 일관성

- [ ] 프로젝트의 폴더 구조 규칙을 따르는지 확인하세요 (예: FSD 아키텍처)
- [ ] 레이어 간 의존성 규칙을 위배하지 않았는지 확인하세요
- [ ] 기존 디자인 패턴과 일관성을 유지하는지 확인하세요
- [ ] 컴포넌트 분리 원칙을 준수했는지 확인하세요

### 5.3 TypeScript 타입 안정성

- [ ] `any` 타입을 남용하지 않았는지 확인하세요
- [ ] 타입 단언(Type Assertion)을 불필요하게 사용하지 않았는지 확인하세요
- [ ] 제네릭을 적절히 활용했는지 확인하세요
- [ ] 타입이 명확하게 정의되었는지 확인하세요

## 6. 보안 검증

### 6.1 환경 변수 및 시크릿

- [ ] 환경 변수에 민감한 정보를 하드코딩하지 않았는지 확인하세요
- [ ] `.env` 파일이 `.gitignore`에 포함되어 있는지 확인하세요
- [ ] API 키, 토큰 등이 코드에 노출되지 않았는지 확인하세요
- [ ] 클라이언트 측 코드에 서버 측 시크릿이 노출되지 않았는지 확인하세요

### 6.2 라이브러리 보안

- [ ] 알려진 취약점이 있는 라이브러리 버전을 사용하지 않았는지 확인하세요
- [ ] 불필요한 권한을 요구하는 라이브러리가 없는지 확인하세요
- [ ] 신뢰할 수 없는 출처의 패키지를 설치하지 않았는지 확인하세요
- [ ] `npm audit` 또는 `pnpm audit`으로 보안 취약점을 확인하세요

### 6.3 코드 보안

- [ ] XSS(Cross-Site Scripting) 취약점이 없는지 확인하세요
- [ ] SQL Injection 가능성이 없는지 확인하세요
- [ ] 사용자 입력을 적절히 검증하고 있는지 확인하세요
- [ ] 인증 및 권한 부여 로직이 올바른지 확인하세요

## 7. 성능 검증

### 7.1 불필요한 리렌더링

- [ ] React 컴포넌트의 불필요한 리렌더링이 발생하지 않는지 확인하세요
- [ ] 메모이제이션이 필요한 곳에 적절히 사용되었는지 확인하세요
- [ ] 의존성 배열이 정확하게 설정되었는지 확인하세요

### 7.2 번들 크기

- [ ] 불필요한 라이브러리를 추가하지 않았는지 확인하세요
- [ ] 트리 쉐이킹이 가능한 방식으로 import 했는지 확인하세요
- [ ] 동적 import를 활용할 수 있는 곳이 있는지 확인하세요

## 8. 문서화 및 커밋 검증

### 8.1 주석 및 문서

- [ ] 복잡한 로직에 적절한 주석이 추가되었는지 확인하세요
- [ ] JSDoc 또는 TSDoc 형식으로 함수 설명이 작성되었는지 확인하세요
- [ ] README, 설정 파일 등 관련 문서가 업데이트되었는지 확인하세요

### 8.2 커밋 메시지

- [ ] `absolute-rule.md`에 정의된 커밋 메시지 형식을 따르는지 확인하세요
- [ ] 변경 사항이 명확히 설명되었는지 확인하세요
- [ ] 하나의 커밋에 너무 많은 변경 사항이 포함되지 않았는지 확인하세요

## 9. TDD 프로세스 준수 확인

> 관련 문서: `absolute-rule.md` - 개발 프로세스 (TDD)

- [ ] 기존 코드 재활용 여부를 최우선으로 검토했는지 확인하세요
- [ ] 새로운 코드 작성 시 테스트 코드를 먼저 작성했는지 확인하세요
- [ ] 테스트가 요구사항을 올바르게 검증하는지 확인하세요
- [ ] 코드가 테스트를 통과한 후 리팩토링을 진행했는지 확인하세요
- [ ] 리팩토링 후 테스트가 여전히 통과하는지 확인했는지 확인하세요

## 10. 전체적인 품질 확인

### 10.1 코드 리뷰 관점

- [ ] 코드가 읽기 쉽고 이해하기 쉬운지 확인하세요
- [ ] 다른 개발자가 유지보수하기 쉬운 코드인지 확인하세요
- [ ] 예외 처리가 적절히 되어 있는지 확인하세요
- [ ] 에러 메시지가 명확하고 도움이 되는지 확인하세요

### 10.2 실행 확인

- [ ] 실제로 코드를 실행하여 동작하는지 확인하세요
- [ ] 개발 서버를 재시작하여 빌드 에러가 없는지 확인하세요
- [ ] 브라우저 콘솔에 에러나 경고가 없는지 확인하세요
- [ ] 의도한 대로 기능이 작동하는지 직접 테스트하세요

## 검증 프로세스

1. **즉시 검증**: AI 답변을 받은 직후, 위 체크리스트를 빠르게 훑어보세요
2. **코드 적용 전**: 코드를 실제로 적용하기 전에 관련 항목을 집중적으로 검증하세요
3. **테스트**: 코드를 적용한 후 반드시 실행하여 동작을 확인하세요
4. **반복**: 문제가 발견되면 AI에게 수정을 요청하고 다시 검증하세요

---

**중요**: 이 체크리스트는 AI 에이전트를 불신하기 위한 것이 아니라, 더 나은 결과물을 만들기 위한 협업 도구입니다. AI와 인간의 강점을 결합하여 최상의 코드를 작성하세요.
